## Known Bugs

**Font override conflict:**
- Symptoms: The Geist font loaded via `next/font/google` in `app/layout.tsx` is set as CSS variables (`--font-geist-sans`, `--font-geist-mono`) and piped into Tailwind's `--font-sans`/`--font-mono` theme tokens in `app/globals.css`. However, `body { font-family: Arial, Helvetica, sans-serif; }` in `app/globals.css` line 25 hard-overrides the font, making the Geist font load wasted.
- Files: `app/globals.css`, `app/layout.tsx`
- Trigger: View any page — body text renders in Arial, not Geist.
- Workaround: Remove the hard-coded `font-family` from the `body` rule, or replace with `font-family: var(--font-sans), sans-serif;`.

## Security Concerns

- [ ] **No security headers configured:** `next.config.ts` is empty — no `Content-Security-Policy`, `X-Frame-Options`, `Strict-Transport-Security`, or other security headers are set. When the app adds real content and blockchain interactions, this is a significant gap.
- [ ] **No middleware:** No `middleware.ts` exists at the project root. When API routes or authenticated pages are added, middleware for auth checks and rate limiting will be needed.
- [ ] **No `.env` file present:** While `.env*` is correctly in `.gitignore`, there is no `.env.example` or `.env.template` to document required environment variables. As Midnight blockchain integration is developed, connection URIs, API keys, and network IDs will be needed.
- [ ] **No Content Security Policy for blockchain SDK:** The `@midnight-ntwrk/*` packages may need WebAssembly (`wasm-eval`) or `connect-src` allowances. CSP must be planned before deployment.
- [ ] **No input validation library installed:** No `zod`, `yup`, `joi`, or similar schema validation library is present. Any future API routes or form handling will need validation.

## Performance Concerns

- [ ] **725MB `node_modules`:** The dependency tree is very large for a scaffold app. The `@midnight-ntwrk/*` packages bring in transitive deps (`compact-runtime`, `onchain-runtime-v3`, `platform-js`, `midnight-js-utils`). Once the app grows, tree-shaking effectiveness should be audited.
- [ ] **Unused dependencies in bundle:** `graphql` (16.13.2) and `graphql-yoga` (5.21.0) are declared as production dependencies but unused. If they remain unused, they may still be analyzed/bundled by Next.js, increasing build time.
- [ ] **No image optimization config:** `next.config.ts` has no `images` configuration. When external images or blockchain-related assets are added, the `remotePatterns` config will be needed.
- [ ] **Google Fonts loaded at runtime:** `app/layout.tsx` uses `next/font/google` for two fonts (Geist, Geist_Mono). If the app doesn't actually use the mono font, it's an unnecessary download.

## Scalability Concerns

- **No API layer exists:** There are no API routes (`app/api/`), no server actions, and no data fetching. The entire app is a static page. All architecture for the polling feature, blockchain interactions, and data flow needs to be built from scratch.
- **No state management:** No client-side state management pattern (React Context, Zustand, Jotai, etc.) is set up. As the polling UI grows, a strategy is needed.
- **No database or data persistence:** No database, ORM, or data layer is configured. The Midnight blockchain dependencies suggest on-chain data, but there's no integration code.
- **Single page app:** Only one route (`/`) exists. The routing structure for polls (create, view, vote, results) is entirely absent.

## Maintainability Concerns

- **No application code to maintain:** The codebase is 100% scaffold boilerplate. The only custom files are `app/page.tsx` (default template), `app/layout.tsx` (default template), and config files. There's no business logic, no components, no utilities, no types.
- **No shared types/interfaces:** No `types/` directory or type definition files for the polling domain.
- **No component library or design system:** No reusable UI components. When the polling UI is built, a consistent component structure should be established from the start.
- **`AGENTS.md` warns about Next.js 16 breaking changes:** The project uses Next.js 16.2.2, which has API/convention differences from prior versions. Developers must consult `node_modules/next/dist/docs/` before implementing features. This is a knowledge risk.
- **No documentation beyond AGENTS.md:** No README with setup instructions, architecture decisions, or project goals.

## Dependency Concerns

- **Unused production dependencies (7 packages):** All `@midnight-ntwrk/*` packages (`compact-js`, `ledger-v8`, `midnight-js-contracts`, `midnight-js-fetch-zk-config-provider`, `midnight-js-indexer-public-data-provider`, `midnight-js-network-id`, `midnight-js-types`) are in `package.json` but imported nowhere. These are presumably planned for future use but add install time, lockfile complexity, and potential version conflicts now.
  - Files: `package.json` lines 12-18
- **Unused GraphQL dependencies (2 packages):** `graphql` and `graphql-yoga` are declared but unused.
  - Files: `package.json` lines 19-20
- **Niche ecosystem risk:** The `@midnight-ntwrk/*` packages are from the Midnight blockchain network — a niche ecosystem. Documentation, community support, and long-term maintenance of these SDKs may be limited compared to mainstream packages.
- **No lock on major versions for dev deps:** `@tailwindcss/postcss`, `eslint`, `tailwindcss`, and `typescript` all use `^` with major version ranges (e.g., `"^4"`, `"^9"`, `"^5"`). This means `bun install` on a fresh machine could pull breaking major version updates.
  - Files: `package.json` lines 26-33

## Code Smells

- **Entire `app/page.tsx` is boilerplate:** 65 lines of default `create-next-app` template with Vercel marketing links. This should be replaced with actual application UI.
  - Files: `app/page.tsx`
- **Magic color values:** `app/page.tsx` uses hard-coded hex colors (`#383838`, `#ccc`, `#1a1a1a`) and opacity values (`black/[.08]`, `white/[.145]`) instead of design tokens.
  - Files: `app/page.tsx` lines 39, 54
- **No test files exist:** Zero `.test.*` or `.spec.*` files. No test runner configured (`jest`, `vitest`, `playwright`, etc.). No `test` script in `package.json`.
- **No error boundary:** No React error boundary component exists. Unhandled runtime errors will crash the UI with no graceful fallback.

## Test Coverage Gaps

**Everything is untested:**
- What's not tested: The entire application — there are zero test files, no test framework installed, and no test script.
- Files: All files in `app/`
- Risk: As features are added, there's no safety net. Regressions will go unnoticed.
- Priority: High — a test framework should be set up before building features.

## Recommended Priorities

1. **Immediate** (before building features):
   - Install and configure a test framework (Vitest recommended for Next.js 16 compatibility)
   - Add a `test` script to `package.json`
   - Remove or comment out unused `@midnight-ntwrk/*` and `graphql` dependencies until they're needed (reduces install time, avoids phantom version conflicts)
   - Replace boilerplate `app/page.tsx` and update metadata in `app/layout.tsx`
   - Fix the font-family conflict in `app/globals.css`

2. **Short-term** (during initial feature development):
   - Create `.env.example` documenting required environment variables
   - Set up security headers in `next.config.ts` (CSP, HSTS, X-Frame-Options)
   - Establish component structure and naming conventions before the first feature PR
   - Add an input validation library (`zod` recommended) for API routes and form data
   - Add a React error boundary component
   - Pin dev dependency major versions to avoid surprise breaks

3. **Long-term** (as the app matures):
   - Audit `@midnight-ntwrk/*` bundle impact — consider dynamic imports or code splitting for blockchain SDK code
   - Add monitoring/error tracking (Sentry or similar)
   - Add middleware for rate limiting and auth when API routes exist
   - Establish CI pipeline with lint + test + build checks
   - Document architecture decisions as the blockchain integration takes shape
