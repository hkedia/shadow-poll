# Phase 10 Plan 01: Invite Code Contract + Service Fixes Summary

**One-liner:** Batch invite code submission via `Vector<10, Bytes<32>>` single-tx circuit + single-use enforcement via boolean flag flip in `cast_invite_vote`.

## What Was Built

Fixed two bugs in the invite-only poll system:

1. **Batch invite code submission** — `add_invite_codes` now accepts `Vector<10, Bytes<32>>` instead of a single `Bytes<32>`, submitting all invite code hashes in one blockchain transaction. The TypeScript service pads the array to exactly 10 elements with zero bytes; the Compact circuit skips zero entries.

2. **Single-use enforcement** — `cast_invite_vote` now checks that the invite code hasn't already been used (`invite_codes.lookup(d_invite_key)` must be `true`) and marks it spent by inserting `false` after a successful vote.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Generic circuit `<#N>` not exportable in Compact 0.30**
- **Found during:** Task 1
- **Issue:** Compact compiler error: "cannot export type-parameterized function (add_invite_codes) from the top level" — the plan's proposed `<#N>` generic circuit syntax compiles internally but cannot be exported.
- **Fix:** Used `Vector<10, Bytes<32>>` (fixed max size = max poll options per D-23) instead of generic `<#N>`. Added zero-byte filtering in the circuit (`if d_code_hash != pad(32, "\x00") { ... }`). TypeScript service in `callAddInviteCodes` pads the caller's array to exactly 10 elements with `new Uint8Array(32)` zero entries.
- **Files modified:** `contracts/src/poll.compact`, `lib/midnight/contract-service.ts`
- **Commit:** aa20128, 0cd6a31
- **Functional equivalence:** Still achieves the plan goal — N hashes in one transaction. The 10-element constraint matches D-23 (max 10 poll options) so callers will never have more than 10 invite codes to submit in one poll creation.
