**1. `cast_invite_vote` circuit** — Vote on an invite-only poll with invite code verification.

```compact
// Cast a vote on an invite-only poll.
// Requirements: CONT-05, CONT-06
// The invite_code is provided as a private witness value — the circuit hashes it
// and checks membership in the invite_codes map. The voter's identity is never
// linked to their choice (D-21). The nullifier prevents double voting (D-51).
export circuit cast_invite_vote(
  poll_id: Bytes<32>,
  option_index: Uint<8>,
  invite_code: Bytes<32>
): [] {
  const d_poll_id = disclose(poll_id);
  const d_option_index = disclose(option_index);
  // invite_code is NOT disclosed — it stays private in the ZK proof

  // Verify poll exists
  assert(polls.member(d_poll_id), "Poll not found");
  const poll_data = polls.lookup(d_poll_id);

  // Only invite-only polls use this circuit (D-52)
  assert(poll_data.poll_type == PollType.invite_only, "Use cast_vote for public polls");

  // Validate option index
  assert(d_option_index < poll_data.option_count, "Invalid option index");

  // Check expiration (D-24)
  const block = disclose(current_block_number());
  assert(block < poll_data.expiration_block, "Poll has expired");

  // Verify invite code via ZK proof (CONT-05)
  // The raw invite_code is private — only its hash is checked against the ledger.
  // The verifier learns "the prover knows a valid invite code" without learning which code.
  const invite_key = derive_invite_key(d_poll_id, invite_code);
  const d_invite_key = disclose(invite_key);
  assert(invite_codes.member(d_invite_key), "Invalid invite code");

  // Duplicate vote prevention via nullifier (CONT-06, D-51)
  const sk = local_secret_key();
  const nullifier = derive_nullifier(d_poll_id, sk);
  const d_nullifier = disclose(nullifier);
  assert(!vote_nullifiers.member(d_nullifier), "Already voted on this poll");
  vote_nullifiers.insert(d_nullifier, true);

  // Increment tally
  const tally_key = derive_tally_key(d_poll_id, d_option_index as Bytes<32>);
  tallies.lookup(tally_key).increment(1);
}
```

Key design points:
- `invite_code` parameter is NOT disclosed — it remains private in the ZK proof. The circuit hashes it internally and only the hash (invite_key) is disclosed for ledger lookup. This is the core ZK privacy: the verifier learns "the prover knows a valid code" without learning which code.
- The nullifier is derived from the voter's secret key, not the invite code — this way each wallet can only vote once, regardless of how many invite codes they hold.

**2. `add_invite_codes` circuit** — Creator adds invite code hashes after poll creation.

This circuit is designed for batch-adding invite code hashes. Due to Compact's limitations with variable-length arrays, we add a fixed batch size. The creator can call this circuit multiple times to add more codes.

Since Compact v0.19 does not support loops or variable-length arrays in circuit parameters, we'll use a fixed-size approach — the creator adds one code hash at a time. This is simpler and more reliable:

```compact
// Add a single invite code hash to an invite-only poll.
// Requirements: POLL-06 (management), D-54 (separate circuit)
// Only the poll creator can add invite codes.
// The code_hash is the pre-computed hash(poll_id, raw_invite_code) — computed
// off-chain by the creator's browser and submitted as a disclosed value.
export circuit add_invite_codes(
  poll_id: Bytes<32>,
  code_hash: Bytes<32>
): [] {
  const d_poll_id = disclose(poll_id);
  const d_code_hash = disclose(code_hash);

  // Verify poll exists
  assert(polls.member(d_poll_id), "Poll not found");
  const poll_data = polls.lookup(d_poll_id);

  // Only invite-only polls accept invite codes
  assert(poll_data.poll_type == PollType.invite_only, "Poll is not invite-only");

  // Only the creator can add invite codes (D-54)
  const sk = local_secret_key();
  const caller_pk = derive_public_key(sk);
  const d_caller_pk = disclose(caller_pk);
  assert(d_caller_pk == poll_data.creator, "Only the poll creator can add invite codes");

  // Store the invite code hash
  invite_codes.insert(d_code_hash, true);
}
```

Key design points:
- `code_hash` is disclosed because it's already a hash — revealing it doesn't leak the raw invite code. The raw code is only known to the creator and the invited voter.
- The creator is verified by deriving their public key from the witness secret key and comparing against `poll_data.creator`.
- One code hash per call. For N invite codes, the creator calls this N times. This avoids Compact's limitations with arrays and keeps the circuit simple.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Spoofing | add_invite_codes (creator check) | mitigate | Circuit asserts caller_pk == poll_data.creator. Attacker cannot forge the public key derivation without the creator's secret key. |
| T-05-02 | Tampering | cast_invite_vote (invite code) | mitigate | Invite code is private in the ZK proof. Hash is checked against on-chain Map. Attacker cannot guess a valid code from the hash (preimage resistance of persistentHash). |
| T-05-03 | Repudiation | vote_nullifiers (duplicate vote) | mitigate | Nullifier is deterministic from wallet + poll. Same wallet always produces same nullifier. Cannot vote twice without a different wallet. |
| T-05-04 | Information Disclosure | cast_invite_vote (invite_code param) | mitigate | invite_code is NOT disclosed — it stays private in the ZK proof. Only the derived hash (invite_key) is disclosed for Map lookup, which reveals nothing about the raw code. |
| T-05-05 | Elevation of Privilege | add_invite_codes (unauthorized) | mitigate | Creator identity verified via derive_public_key(witness secret_key) == poll_data.creator. Non-creator cannot pass this assertion. |
