**Step 1: Create `contracts/scripts/compile.sh`**

Create directory `contracts/scripts/` and write the compilation script.

The script must:
1. Set `set -euo pipefail` for safety
2. Define paths relative to repo root (detect via `SCRIPT_DIR` and navigate up)
3. Define variables:
   - `CONTRACT_SRC="contracts/src/poll.compact"`
   - `MANAGED_DIR="contracts/managed"`
   - `ZK_KEYS_DIR="public/zk-keys"`
   - `COMPACT_VERSION="+0.28.0"` (pinned version matching the installed SDK — use the version from Plan 01)
4. Clean previous output: `rm -rf "$MANAGED_DIR"` and `mkdir -p "$MANAGED_DIR"`
5. Create ZK keys output dir: `mkdir -p "$ZK_KEYS_DIR"`
6. Run full compilation (WITH ZK key generation — no `--skip-zk` flag):
   ```bash
   compact compile $COMPACT_VERSION "$CONTRACT_SRC" "$MANAGED_DIR"
   ```
7. After compilation, copy ZK key files to public directory:
   ```bash
   # Copy all proving (.prover) and verifying (.verifier) key files
   find "$MANAGED_DIR" -name "*.prover" -o -name "*.verifier" | while read -r file; do
     cp "$file" "$ZK_KEYS_DIR/"
   done
   ```
   NOTE: The exact file extensions for ZK keys depend on the Compact compiler output. The compiler generates `.prover` and `.verifier` files (or possibly `.pk` / `.vk` files). After the first compilation run, check `contracts/managed/` to see what file types were generated and adjust the copy pattern accordingly. If the compiler generates different extensions (e.g., `.zkir`-based keys), copy those instead. The key insight is: any file that the `@midnight-ntwrk/midnight-js-fetch-zk-config-provider` needs at runtime should be copied to `public/zk-keys/`.
8. Print success message with list of copied files.

Make the script executable: `chmod +x contracts/scripts/compile.sh`

**Step 1: Create `public/zk-keys/.gitkeep`**

**Step 2: Update `next.config.ts` to add CORS headers for ZK keys (per CONT-04)**

The `@midnight-ntwrk/midnight-js-fetch-zk-config-provider` fetches ZK proving/verifying keys via HTTP. When the dApp serves these from its own `public/` folder, CORS headers must allow the browser to read them.

Add a `headers()` async function to the Next.js config that serves CORS headers for `/zk-keys/*` paths:

```typescript
const nextConfig: NextConfig = {
  serverExternalPackages: MIDNIGHT_EXTERNAL_PACKAGES,
  turbopack: {
    resolveAlias: Object.fromEntries(
      MIDNIGHT_EXTERNAL_PACKAGES.map((pkg) => [pkg, "./lib/midnight-sdk-stub.ts"])
    ),
  },
  async headers() {
    return [
      {
        source: "/zk-keys/:path*",
        headers: [
          { key: "Access-Control-Allow-Origin", value: "*" },
          { key: "Access-Control-Allow-Methods", value: "GET, OPTIONS" },
          { key: "Access-Control-Allow-Headers", value: "Content-Type" },
          { key: "Cache-Control", value: "public, max-age=31536000, immutable" },
        ],
      },
    ];
  },
};
```

The `Cache-Control: immutable` header is appropriate because ZK keys are deterministic outputs of compilation — they change only when the contract changes, at which point the filenames change too.

**Important:** Preserve ALL existing config (serverExternalPackages, turbopack resolveAlias). Only ADD the `headers()` function.

## Trust Boundaries

|----------|-------------|
| public/ → browser | ZK key files served as static assets — must not leak contract internals |
| compile script → filesystem | Script writes to managed/ and public/ — path traversal risk minimal (hardcoded paths) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07 | Information Disclosure | ZK keys in public/ | accept | Proving/verifying keys are meant to be public — they enable ZK proof generation/verification. No secrets are in key files. |
| T-02-08 | Tampering | compile.sh injection | accept | Script runs locally by developer; no untrusted input. `set -euo pipefail` prevents silent failures. |
| T-02-09 | Denial of Service | CORS wildcard origin | accept | Using `*` for CORS is safe here — ZK keys are public data, no authentication needed. Restricting origin would break legitimate dApp usage. |
