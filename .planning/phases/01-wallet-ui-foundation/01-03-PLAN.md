**Step 2: Update next.config.ts for WASM compatibility.**

The Midnight SDK packages (`@midnight-ntwrk/ledger-v8`, `@midnight-ntwrk/compact-js`, etc.) use WASM. Next.js needs explicit config to handle them:

```typescript
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  experimental: {
    // Required for Midnight SDK WASM packages
    serverComponentsExternalPackages: [
      "@midnight-ntwrk/compact-js",
      "@midnight-ntwrk/ledger-v8",
      "@midnight-ntwrk/midnight-js-contracts",
      "@midnight-ntwrk/midnight-js-fetch-zk-config-provider",
      "@midnight-ntwrk/midnight-js-indexer-public-data-provider",
      "@midnight-ntwrk/midnight-js-network-id",
      "@midnight-ntwrk/midnight-js-types",
    ],
  },
  webpack: (config, { isServer }) => {
    // Handle WASM files from Midnight SDK
    config.experiments = {
      ...config.experiments,
      asyncWebAssembly: true,
      layers: true,
    };

    if (!isServer) {
      // Prevent WASM packages from being bundled on the client directly
      // They are dynamically imported in lib/midnight/providers.ts
      config.resolve.fallback = {
        ...config.resolve.fallback,
        fs: false,
        path: false,
        crypto: false,
      };
    }

    return config;
  },
};

export default nextConfig;
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| WalletProvider → children | All wallet state flows down through React context — no external data crosses here |
| app/page.tsx → wallet status | Page renders based on status from context — no user-controlled input |
| next.config.ts webpack config | Build-time only; WASM fallbacks are defensive, not security-critical |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | app/page.tsx status check | accept | `status === "not_detected"` is derived from `window.midnight` detection — not user-controlled input. The install prompt is purely UI, no data flows to backend |
| T-03-02 | Information Disclosure | Hero section connected state | accept | Connected state shows only the "Secure Anonymous Voting Active" badge — no address or wallet data is exposed in the hero section itself (address is in header only) |
| T-03-03 | Denial of Service | WASM dynamic imports in providers.ts | accept | Dynamic imports are lazy; if WASM packages fail to load at runtime, the error is caught in connectInternal() and shown as error state. No crash. |
| T-03-04 | Elevation of Privilege | WalletProvider context access | accept | WalletProvider exposes connect/disconnect functions — these only trigger wallet API calls that require user approval in the extension. No privileged server-side actions. |
</threat_model>
